# 第四章：Android 性能优化

性能优化是高级 Android 面试的分水岭。中级工程师往往只会说“卡就优化布局、内存大就查泄漏”，而高级面试希望你能建立指标、工具、原因和方案之间的闭环。本章覆盖启动速度、内存、渲染、网络、包体积、电量与性能分析工具。

```text
用户感知性能
├─ 启动快不快
├─ 列表滑不滑
├─ 图片出不出
├─ 电量耗不耗
└─ App 会不会被杀 / ANR / OOM
```

## 1. 启动优化

### 1.
**题目**：冷启动（Cold Start）、温启动（Warm Start）、热启动（Hot Start）分别指什么？

**考察要点**：进程状态、Activity 状态、优化重点。

**参考答案**：冷启动是进程不存在，需要从 Zygote fork、初始化 Application、创建首个 Activity；温启动是进程还在，但 Activity 需要重新创建；热启动通常指 Activity 已在内存中，只是从后台快速回到前台。三者中冷启动最贵，也是优化重点。面试里最好补充：不同厂商统计口径略有差异，所以讨论前应先统一指标定义。

### 2.
**题目**：App Startup library 的价值是什么？

**考察要点**：初始化治理、依赖声明、延迟初始化。

**参考答案**：App Startup library 用于集中声明应用启动初始化项及其依赖关系，替代散落的 `ContentProvider` 初始化。它的价值在于初始化可观测、可排序、可延迟，而不是“多一个库而已”。答题时应强调：真正的优化不是把所有东西都放进 App Startup，而是借此识别哪些初始化必须首屏前完成，哪些可以后移。

### 3.
**题目**：什么是 Baseline Profiles？为什么它很重要？

**考察要点**：AOT、热点代码、安装后性能。

**参考答案**：Baseline Profiles 通过提前告诉 ART 哪些方法和路径是应用高频热点，让用户安装后就能获得更好的预编译效果，从而改善启动和关键交互性能。它的重要性在于：现代 Android 性能优化不只靠代码瘦身，还要利用平台编译链。面试里如果你能提到它对启动、列表滚动、页面切换都有帮助，会更完整。

### 4.
**题目**：启动追踪通常怎么做？

**考察要点**：Tracing、Perfetto、宏基准。

**参考答案**：常见方法包括 Android Studio Profiler、Perfetto/System Trace、Macrobenchmark 和自定义埋点。思路是先定义从进程创建到首帧可交互的时间边界，再拆分 Application 初始化、首屏数据加载、首帧渲染等阶段。高分答案不是背工具名，而是能说明如何从 trace 里定位主线程阻塞、Binder 调用和 IO 热点。

## 2. 内存优化

### 5.
**题目**：Android 常见内存泄漏有哪些？

**考察要点**：内部类、静态引用、Handler、监听器。

**参考答案**：典型泄漏包括：非静态内部类持有外部 Activity、单例/静态对象持有 `Context`、Handler/延迟任务未释放、匿名监听器或回调未注销、Bitmap 生命周期失控。面试中不要只列清单，最好补一句根因：对象生命周期长于其应持有的宿主，导致 GC 无法回收。

### 6.
**题目**：LeakCanary 的工作流是什么？

**考察要点**：监控对象、弱引用、堆分析。

**参考答案**：LeakCanary 会在关键组件销毁后监控其弱引用是否被回收；若长时间未回收，就触发 heap dump 并分析引用链，告诉你是谁仍然持有对象。它最大的价值是把“怀疑泄漏”变成可定位的证据。面试时补充“线上环境要控制 dump 成本，通常开发和测试阶段使用最频繁”会更贴近工程实践。

### 7.
**题目**：什么是 GC Pressure，为什么它会造成卡顿？

**考察要点**：频繁分配、短命对象、停顿。

**参考答案**：GC Pressure 指对象分配过于频繁，导致垃圾回收器需要更频繁工作。即便单次 GC 不长，持续的小停顿也会抢占主线程时间片，造成掉帧和列表不流畅。优化思路通常是减少临时对象、复用缓冲区、避免在高频路径做字符串拼接或装箱。面试里强调“不是只有 OOM 才算内存问题”很重要。

### 8.
**题目**：如何用 Memory Profiler 定位问题？

**考察要点**：分配跟踪、对象保留、heap dump。

**参考答案**：先看整体内存曲线是否持续爬升，再结合对象分配记录观察高频分配点，必要时做 heap dump 看大对象与引用链。若怀疑泄漏，重点看 Activity、Fragment、Adapter、Bitmap 是否在页面退出后仍存活；若怀疑抖动，则关注短时间内是否有大量临时对象产生。好答案会区分“泄漏”和“抖动”是两类不同问题。

## 3. UI 渲染优化

### 9.
**题目**：为什么 60fps 意味着每帧只有约 16.6ms？

**考察要点**：帧预算、主线程、掉帧概念。

**参考答案**：60fps 表示每秒渲染 60 帧，因此一帧预算约为 1000/60 ≈ 16.6ms。这个时间不仅包含你自己的代码，还包括输入处理、动画、布局、绘制、GPU 提交等全部流程。只要任一阶段超时，就可能发生 jank。面试时建议补充 120Hz 设备的预算更紧，大约只有 8.3ms。

### 10.
**题目**：什么是 GPU Overdraw？如何优化？

**考察要点**：重复绘制、层叠背景、可视化工具。

**参考答案**：Overdraw 指同一像素在一帧中被重复绘制多次，比如多层背景、全屏遮罩、不可见但仍在绘制的层。它会增加 GPU 负担，严重时影响滚动流畅度。优化方式包括移除无意义背景、减少嵌套层、避免透明大面积叠加，并使用开发者选项或 Layout Inspector 辅助观察。

### 11.
**题目**：传统 View 布局层级过深有什么问题？

**考察要点**：多次 measure/layout、inflate 成本。

**参考答案**：层级越深，测量和布局递归成本越高，inflate 也更慢；某些 `wrap_content`、权重布局还会触发额外测量。优化思路通常是扁平化布局、使用 `ConstraintLayout`、合并容器、延迟加载不必要区域。面试里要补充：不是“节点越少越好”，而是综合可读性与性能做权衡。

### 12.
**题目**：Compose 性能问题和 View 性能问题有什么不同？

**考察要点**：重组、状态粒度、对象稳定性。

**参考答案**：View 体系常见问题是层级深、重复 `requestLayout`、onDraw 过重；Compose 更常见的是无意义重组、unstable 参数、在组合中频繁创建对象。Compose 并不天然比 View 慢或快，关键在于你是否遵守其状态与稳定性模型。面试里要体现的是“定位方式不同”，而不是简单站队。

### 13.
**题目**：如何做帧级性能分析？

**考察要点**：Jank、Frame Timeline、Perfetto。

**参考答案**：可以通过 JankStats、Perfetto 的 Frame Timeline、Profile GPU Rendering 等工具观察慢帧。分析重点是找出掉帧发生在输入、布局、绘制还是 GPU 提交阶段，再把问题映射到具体代码路径。高分回答常会提到“先复现稳定场景，再抓 trace，对比优化前后”，体现实验思维。

## 4. 网络与图片性能

### 14.
**题目**：OkHttp 拦截器（Interceptor）有哪些类型，常见用途是什么？

**考察要点**：应用拦截器、网络拦截器、日志、鉴权。

**参考答案**：OkHttp 主要有应用拦截器和网络拦截器。前者更适合统一 header、签名、重试、日志与业务级错误包装；后者能看到更接近网络层的行为，如重定向、缓存命中后的细节。面试时别只说“打印日志”，而要说明拦截器链是网络治理的关键切面。

### 15.
**题目**：HTTP 缓存、连接池、HTTP/2 为什么能提升移动端性能？

**考察要点**：减少往返、复用连接、多路复用。

**参考答案**：缓存能减少重复请求和弱网等待；连接池避免每次重新建 TCP/TLS；HTTP/2 支持多路复用，多个请求共享一条连接，降低队头阻塞。移动端网络优化的本质是减少握手、减少字节、减少等待。优秀回答还会提到“缓存策略必须与业务一致，不能为了快而破坏数据正确性”。

### 16.
**题目**：图片加载框架为什么一定要做缓存和采样？

**考察要点**：内存占用、解码成本、滚动流畅度。

**参考答案**：原图解码成本高、占内存大，如果不按目标尺寸采样，极易造成卡顿甚至 OOM。缓存则能避免重复下载和重复解码，是列表体验的基础。面试里可提到 Coil/Glide 的典型链路：请求合并、内存缓存、磁盘缓存、解码、变换、结果回填，这能体现你理解的是系统而不是单个 API。

## 5. 包体积与电量

### 17.
**题目**：R8/ProGuard、资源压缩（Resource Shrinking）、App Bundle 的作用分别是什么？

**考察要点**：代码压缩、无用资源移除、按设备分发。

**参考答案**：R8/ProGuard 负责代码压缩、混淆、优化；资源压缩会剔除无引用资源；App Bundle 让商店按设备 ABI、语言、密度动态分发更小安装包。体积优化的核心收益不仅是下载快，还包括安装快、冷启动快和磁盘占用低。高分回答会提醒：混淆配置和反射/序列化兼容要谨慎验证。

### 18.
**题目**：动态特性模块（Dynamic Feature Modules）适合什么场景？

**考察要点**：按需下载、体积控制、复杂度权衡。

**参考答案**：动态特性模块适合低频功能、国家/地区差异功能、大型编辑器或 AR 能力等体积昂贵功能。它能减少首包大小，但会带来安装时机、资源访问、导航和测试复杂度。面试答题时要体现权衡：不是所有功能都值得拆成动态模块。

### 19.
**题目**：Doze mode、JobScheduler、WorkManager、前台服务该如何选择？

**考察要点**：电量限制、任务时效性、系统策略。

**参考答案**：Doze mode 下系统会严格限制后台网络和唤醒；JobScheduler 是系统级调度基础设施；WorkManager 是跨版本统一封装；前台服务适合用户明确感知、必须持续进行的任务，如导航、录音、通话。面试要说清：Android 12+ 对后台启动和前台服务限制更严，设计方案必须尊重平台规则，而不是想着绕过。

### 20.
**题目**：定位（Location）相关能力如何做电量优化？

**考察要点**：精度、频率、批量上报、场景感知。

**参考答案**：核心策略是按业务场景动态调整精度和采样频率，优先使用融合定位能力，必要时采用批量上报和地理围栏。不要在后台长期高精度拉定位，除非业务强依赖且用户明确知情。优秀回答会把“技术方案、权限合规、用户感知”一起考虑。

### 21.
**题目**：Perfetto、Systrace、Layout Inspector、Android Profiler 应该怎么分工使用？

**考察要点**：工具边界、定位维度。

**参考答案**：Perfetto/Systrace 更适合系统级时间线分析，尤其是启动、掉帧、Binder、线程调度；Layout Inspector 适合看布局层级、Compose 结构、重组信息；Android Profiler 则覆盖 CPU、内存、网络等基础监控。真正高效的做法不是盲抓所有工具，而是先根据症状确定维度，再选最合适的工具切入。


### 22.
**题目**：列表图片和数据为什么要做预取（Prefetch）？

**考察要点**：滚动体验、命中率、过度预取风险。

**参考答案**：预取的目标是在用户真正滑到目标内容前，把即将需要的数据、图片或布局准备好，从而减少白屏和抖动。RecyclerView、Paging 3、图片框架都可以做不同层面的预取，但要控制窗口大小，否则会带来额外 IO、内存与电量开销。面试时可以强调：预取不是越激进越好，而是根据滚动速度、网络条件和缓存命中率动态权衡。

### 23.
**题目**：如何判断一次性能优化是否真的有效？

**考察要点**：基线、可重复实验、统计口径。

**参考答案**：必须先建立基线，再在可重复场景中采集优化前后的启动时延、慢帧比例、内存峰值、网络耗时等指标，避免“体感变快”式结论。理想做法是固定测试机型、构建方式、网络条件和样本次数，必要时引入 Macrobenchmark 或自定义埋点做量化回归。高级面试特别看重这一点，因为真正的性能工作不是猜，而是测量、验证和回归防退化。

### 24.
**题目**：为什么性能优化要分“症状、指标、根因、验证”四步讲？

**考察要点**：面试表达、工程方法、避免拍脑袋。

**参考答案**：因为性能问题最怕只描述症状却没有证据。看到“卡、慢、耗电”只是起点，还需要把现象量化成指标，再通过 trace、profiler、日志等手段定位根因，最后回到实验验证优化是否成立。面试官真正想听到的是你的分析方法，而不是一串零散技巧。

## 本章要点

1. 性能优化要建立“指标—工具—原因—方案—验证”的闭环。
2. 启动优化、内存优化、渲染优化、网络优化、电量优化彼此相关，不能割裂看。
3. Compose 与 View 的问题模型不同，定位方法也不同。
4. 包体积、电量和网络策略都受平台规则影响，工程实践必须尊重 Android 系统约束。
5. 高级面试更看重你如何分析和验证，而不是记住多少优化术语。
